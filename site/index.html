A Human-Centered Research OS

AI handles execution.
Humans own the direction.

AutoR is a terminal-first research harness with a local browser Studio. Eight stages. A human approval gate between each one. Every run leaves behind prompts, logs, approved summaries, code, data, figures, LaTeX sources, and a compiled paper — all under runs/<run_id>/.

At a glance

Six guarantees, not marketing claims.

These are the properties AutoR enforces at the workflow level — not aspirations, not demo scripts. If a run violates any of them, it doesn't pass.

01
Human approval by default
Every stage boundary pauses for a human. Nothing advances silently.
02
Approved summaries are the only memory
Cross-stage context comes from human-approved summaries — never raw agent chatter.
03
Runs are isolated and resumable
Every run lives in its own directory. Stop, resume, redo a stage, roll back downstream.
04
Later stages produce real artifacts
Data, results, figures, LaTeX, PDF — not markdown-only theater.
05
Coding agent as execution layer
Claude Code or Codex CLI handle execution. AutoR is the control loop above.
06
Optional strict reviewer
--full-auto replaces the human gate with a strict reviewer agent for unattended runs.
Positioning

Most autoresearch systems optimize for autonomy.
AutoR takes a different position.

The interesting question isn't "can the agent go further on its own." It's "can a human verify every part of it."

AutoR is

  • A fixed 8-stage pipeline with a human gate between each stage.
  • A run directory that is a self-describing research artifact.
  • A terminal CLI and a local browser Studio, sharing the same on-disk model.
  • Artifact-backed: data, results, figures, LaTeX, compiled PDF — all on disk.
  • Resumable: --resume-run, --redo-stage, --rollback-stage.

AutoR is not

  • A chat demo.
  • A generic multi-agent framework.
  • A markdown-only research toy.
  • A hosted service — it runs local.
  • A paper-shaped polish machine without real experiments.

The Approval Loop

Six options at every gate.
Refine, redirect, or advance.

When a stage finishes a draft, the agent suggests three refinements. You can pick one, write your own feedback, approve, or abort. The stage conversation stays alive across refinement loops — feedback doesn't reset the session.

python main.py --goal "Sparse MoE transfer"

1·2·3
AI refinement
Continue the same stage session with one of the agent's own suggestions.
4
Custom feedback
Write what you want changed in plain language. Feedback is woven into the next attempt's prompt — not a separate Claude roundtrip.
5
Approve
Promote the stage summary into memory.md. Advance to the next stage.
6
Abort
Stop the run. State is preserved on disk — resume later with --resume-run.
AutoR Studio

Same run, same artifacts,
now in a browser.

Studio is a local web UI over the same run directories the CLI writes. Launch it with python studio.py and open http://127.0.0.1:8000/studio/. It streams real Claude tool calls from logs_raw.jsonl and shares state with the terminal.

Live stage monitoring
Pill strip for all 8 stages. Streaming session trace parses Claude tool calls in real time.
Feedback woven into prompts
Feedback is not a chat bubble — it lands in the first attempt's prompt of the next run.
Restart-safe resume
Stop the server and come back. Approve/Feedback lazy-resumes the on-disk run without re-running finished stages.
Paper preview
Compiled PDF, LaTeX sources, and build log in one tab. Overleaf-style.
Versions timeline
Full checkpoint and attempt timeline for every stage. Inspect any past draft.
Same on-disk model
Studio reads and writes the same runs/ directory. CLI and browser stay in lockstep.

Quick start

Four steps. Local. No hosted service.

Python 3.10+, the Claude CLI on PATH, and a few minutes. No hosted backend. No requirements.txt to install from — the runtime dependencies are your agent CLI plus local TeX (optional, for Stage 07).

1
Clone the repo
Python 3.10+ required. git clone https://github.com/jiangyi01/AutoR.git && cd AutoR.
2
Install an agent CLI
Claude CLI (claude) on PATH, or Codex CLI for --operator codex. Local TeX helps for Stage 07 but isn't required for smoke tests.
3
Start a run
Run python main.py --goal "your research thesis". The pipeline pauses after each stage for human approval. Use --fake-operator for a no-agent smoke test.
4
Open Studio (optional)
Run python studio.py and open http://127.0.0.1:8000/studio/ for the browser workspace over the same run directory.

Roadmap

What's shipped, what's next.

We build toward a real research workflow — not more demo surface. The most valuable next steps are the ones that make AutoR tighter, not broader.

Shipped

  • Stage-local continuation sessions — one Claude conversation per stage, reused for refinement loops.
  • Artifact-level validation — enforces data, results, figures, LaTeX, PDF, and review assets at the right stages.
  • Machine-readable run manifest — single source of truth for stage status, approval state, and stale dependencies.
  • Artifact index — artifact_index.json indexes data/, results/, and figures/.
  • AutoR Studio — local browser UI over the same run-based workflow.
  • --full-auto mode — optional strict reviewer agent for unattended runs.
  • Codex backend — Codex CLI alongside Claude via --operator codex.

Next up

  • Deeper cross-stage rollback — stronger downstream stale-state handling after earlier-stage changes.
  • Continuation handoff compression — short machine-generated handoff files to keep long runs stable.
  • Richer artifact indexing — better metadata for data/, results/, figures/, and writing/.
  • Stronger automated tests — cover repair, resume fallback, artifact validation, approval-loop correctness.
  • Writing pipeline hardening — more reliable LaTeX, bibliography, and venue-aware layouts.
  • Review & dissemination packages — readiness checklists, release notes, reviewer bundles.
FAQ

Common questions.

Is AutoR an autonomous AI scientist?

No. AutoR's whole position is against autonomy-first autoresearch. AI handles execution; humans retain approval and direction at every stage boundary. You can turn on --full-auto for unattended runs, but the default is human-reviewed.

What does AutoR actually execute with?

A coding agent CLI — Claude Code or Codex CLI — is the execution layer. AutoR is the research control loop above it: stage progression, validation, repair, recovery, and human checkpoints.

Does it run locally?

Yes. No hosted service. Runs live in runs/<run_id>/ on your disk. Studio is a local web server on 127.0.0.1.

Can I stop a run and come back to it?

Yes. Resume with python main.py --resume-run <id> (or latest). You can also --redo-stage or --rollback-stage. Studio lazy-resumes the same on-disk run.

What counts as "passing validation"?

Both the stage summary's shape (objective, what I did, key results, files produced, 3 refinement suggestions, 6 user options — no [TODO] placeholders) and the real artifacts on disk (data, results, figures, LaTeX, PDF, review assets at the appropriate stages). A run with only markdown notes does not pass.

How is feedback handled?

Custom feedback from option 4 is woven into the first attempt's prompt of the next run. It's not a separate chat turn. The stage conversation stays alive across refinement loops.

Where do the references come from?

Studio borrows the shape of calm long-process tools — VS Code's panel-first layout, Overleaf's source↔PDF pairing, Linear's keyboardy calm, Cursor's inline agent trace, NotebookLM's evidence-grounded conversation. Not chat apps, not notebooks.
